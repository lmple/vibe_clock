# Implementation Plan: Two-Part Report Layout

**Branch**: `004-report-layout` | **Date**: 2026-03-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-report-layout/spec.md`

## Summary

Restructure the `report` command output from a project-grouped layout with truncated descriptions to a two-part layout: (1) a project summary table showing each project and its total duration, and (2) a per-day breakdown showing complete, untruncated task entries grouped by date in ascending order. The same structure is applied to both terminal and PDF output. No CLI interface changes are required.

## Design Decisions

### 1. `Report` struct refactoring

Remove `project_sections: Vec<ProjectSection>` (which conflated summary and detail). Replace with:
- `project_summaries: Vec<ProjectSummary>` — name + total only, for the summary table
- `daily_sections: Vec<DailySection>` — date + `Vec<DailyEntry>`, for the per-day breakdown

`DailyEntry` carries a resolved `project_name: String` alongside the `TaskEntry`, so the renderers don't need to re-resolve project names.

### 2. Terminal description wrapping

Helper `print_row_with_wrapped_description(id, project, description, start, end, duration, desc_width)` splits the description into chunks of `desc_width` characters. The first chunk is printed on the main row; each additional chunk is printed on a continuation row with all other columns left blank. Column widths:

```
ID(6) | Project(18) | Description(40) | Start(7) | End(7) | Duration(8)
```

### 3. PDF description wrapping

Change `Text::new(desc)` to `Paragraph::new(desc)` in the daily detail table cells. `genpdfi` `Paragraph` elements auto-wrap within their allocated column width. No other PDF changes.

### 4. Date ordering

`generate_report()` uses a `BTreeMap<NaiveDate, Vec<DailyEntry>>` internally (BTreeMap sorts keys ascending), then converts to `Vec<DailySection>`. This gives oldest-first ordering for free.

### 5. Task ordering within a day

Entries within each `DailySection` are sorted by `(start_time_key, id)` where `start_time_key = entry.task.start_time.unwrap_or(NaiveDateTime::MAX)`. This places timed tasks first (ascending), then untimed tasks in insertion order.
